# Paper Auto-Peel + Stale Cull + Live SnapTrade Dashboard

**Status:** Draft 2026-05-26. Awaiting review.
**Author:** session-driven spec.

## Problem

Two unrelated but related gaps surfaced in the 2026-05-26 paper book audit:

### Gap A — Paper engine never executes the peel

`agents/trading/rules.py` (shared rules engine) **fires** `target1` / `target2`
events when paper positions cross +20% / +40%. The Slack message says
*"consider selling half, move stop to breakeven."*

But `agents/trading/alpaca_monitor.py` only places SELL orders on three
conditions:
1. Stop price hit
2. Stage 3 / 4 in screener (collapse)
3. ATR exit (-1.5x mult from SMA50)

T1 / T2 hits are alert-only. The engine never auto-peels. Result today
(2026-05-26 paper book):

- SNDK +24.1% — past T1 $1577 → still 21 sh, no peel placed
- LSCC +22.0% — past T1 → still 123 sh
- MTSI +16.9% — approaching T1, no plan to peel half when it crosses

The 2026-05-10 ATR-tiered-T1 spec (`atr-tiered-t1-trim.md`) already mandated
this for live. Paper was excluded then because the engine didn't auto-act.
Now it should.

### Gap B — Stale entries rot in the book

Three paper positions today are 14+ days post-entry, flat or slightly down,
with no momentum and no stop-hit:

- VRT -1.4%, $26,677 tied, entered 2026-05-08
- AHR -0.1%, $29,970 tied, entered ~mid-May
- RSI -2.5%, $14,278 tied, entered 2026-05-06

Combined $70,925 of buying power locked in dead positions. The engine
doesn't cull them because no exit rule fires. They drift forever.

### Gap C — No live SnapTrade dashboard

A working paper dashboard exists at `data/claude_portfolio.html`
(generated by `utils/generators/generate_portfolio.py` on every paper monitor
run). It pulls Alpaca account + positions + history and renders a
light-theme HTML page deployed via GitHub Pages.

The live SnapTrade book has no equivalent. Position questions on live force
ad-hoc `python -c` invocations and stale `positions.json` reads. The
`/pos-review` skill prints to terminal but isn't reviewable as a permanent
URL.

## Solution

Three changes, each independent and small:

### 1. T1 / T2 auto-peel in paper monitor

In `agents/trading/alpaca_monitor.py`, after the existing stop / Stage /
ATR-exit checks, add a new block that consumes T1 / T2 events from
`apply_position_rules()` and places SELL orders:

```python
for event in events:
    if event["kind"] == "target1" and not stop_info.get("t1_peeled"):
        peel_qty = int(qty / 2)
        if peel_qty >= 1:
            result = place_sell(ticker, str(peel_qty))
            if result:
                stop_info["t1_peeled"] = True
                stop_info["stop_price"] = max(stop_info["stop_price"], entry_price * 1.005)
                slack_alert(f":dart: [PAPER] T1 AUTO-PEEL {ticker} sold {peel_qty} sh · "
                            f"stop → breakeven $entry*1.005 · {qty - peel_qty} sh continue")
    elif event["kind"] == "target2" and not stop_info.get("t2_peeled"):
        peel_qty = int(qty / 2)  # half of remaining = 25% of original
        if peel_qty >= 1:
            result = place_sell(ticker, str(peel_qty))
            if result:
                stop_info["t2_peeled"] = True
                slack_alert(f":dart::dart: [PAPER] T2 AUTO-PEEL {ticker} sold {peel_qty} sh · {qty - peel_qty} runner")
```

New `paper_stops.json` fields: `t1_peeled` (bool), `t2_peeled` (bool).
Backfill = False on existing.

**Edge cases:**
- If `qty == 1`, skip peel (can't split 1 sh).
- If `peel_qty * price < $50`, log and skip (don't pay commission on tiny lots).
- Slack alert must respect `[PAPER]` prefix per existing convention.

**Test:**
- Unit test in `tests/test_paper_auto_peel.py` — mock `place_sell`, feed
  rules events, assert sell placed + `t1_peeled` set + stop raised.

### 2. Stale-entry rule — applies to BOTH paper (auto-cull) and live (critical alert)

**Rationale extension:** In a flying market, dead capital is the biggest leak.
A position that's been flat for 14 days while leaders run +20-30% is a
~25% opportunity cost on that slot. Worse than a small loss. Applies
equally to paper and live — the cost of capital locked in dud is the same.

**Paper (auto-execute):** In `agents/trading/alpaca_monitor.py`, after T1/T2 block, add a stale check:

```python
if stop_info.get("entry_date"):
    days_open = (datetime.today() - parse(stop_info["entry_date"])).days
    peak_gain = stop_info.get("peak_gain_pct", 0)
    if days_open >= STALE_DAYS and peak_gain < STALE_PEAK_THRESHOLD:
        # Stale: been flat too long, free the capital
        result = place_sell(ticker, str(int(qty)))
        if result:
            slack_alert(f":zzz: [PAPER] STALE CULL {ticker} — {days_open}d, "
                        f"peak only +{peak_gain:.1f}% · sold {qty} sh")
            stop_info["status"] = "closed_stale"
```

Constants:
- `STALE_DAYS = 14` (two weeks of no progress)
- `STALE_PEAK_THRESHOLD = 4.0` (peak gain < +4% = nothing happened)

**Rationale:** every trade gets 14 days to prove itself. If peak never
crossed +4% in two weeks, the setup didn't work — free the capital for the
next signal. Doesn't fire on losses (those go through stop logic) — fires
on the silent failures that just drift.

**Live (alert-only, no execute):** In `agents/trading/position_monitor.py`,
after existing stop/T1/T2 events emit, add a stale check that fires a
`💤 STALE` Slack alert when same threshold hit. Append to
`CRITICAL_EVENT_KINDS` so it routes to the critical Slack post (every 30 min
during market hours), not the book digest. New event kind: `"stale_entry"`.
Message format:
`":zzz: STALE — {ticker} {days_open}d open, peak only +{peak:.1f}% · "
 "consider cutting (capital opportunity cost)"`.

Dedup: once `stale_alerted_date` is set on the position for the day, no
re-alert that day. Re-arms next trading day.

**Test:**
- Paper: unit test asserting cull fires at day 14 with peak < 4%, doesn't
  fire at day 13 or with peak > 4%.
- Live: unit test asserting alert event emitted (no sell call), correct
  kind, dedup respected.

### 3. Live SnapTrade dashboard

New generator: `utils/generators/generate_live_portfolio.py`. Mirrors
`generate_portfolio.py` structure but:

- **Data source:** SnapTrade `/accounts` + `/positions` + `/balances` via
  `agents/trading/position_monitor.snaptrade_get`. Env: `SNAPTRADE_*`.
- **Output path:** `data/live_portfolio.html`
- **URL:** `https://ananthsrinivasan.github.io/finviz-screener-agent/data/live_portfolio.html`
- **Refresh:** runs inside `agents/trading/position_monitor.py` (3x daily
  book run cron + every 30 min during market hours via critical run).
- **Theme:** light (per `feedback_light_theme.md`).

**Page sections:**

1. **Account header** — equity, cash, buying power, MV, P/L today,
   leverage (debt/equity %).
2. **Positions table** sorted by MV desc, columns:
   `TKR · Sh · AvgCost · Live · Δ% · $P/L · MV · %Book · ATR% · S20% · Stage · Verdict`
3. **Verdict logic** (same as `/pos-review` skill):
   - 🚨 CUT if Δ% ≤ -5
   - 💀 dead weight if 0 ≤ Δ% < 1 AND MV > $5k
   - 🟡 sleeping if 1 ≤ Δ% < 2
   - ⚠ peel ⅓ if Δ% ≥ 7 AND ATR > 7%
   - 🟢 trail tighter if 10 ≤ Δ% < 20
   - 💰 PEEL ½ if Δ% ≥ 20
   - ✅ working if 5 ≤ Δ% < 10
   - hold if 2 ≤ Δ% < 5
4. **Action summary** — count of peel candidates, cuts, dead weight.
5. **Index link** — added to `index.html` (next to existing paper portfolio link).

**Implementation notes:**
- Reuse `_fmt_money` / `_fmt_pct` from `generate_portfolio.py`.
- Color bands: green for working/peel, amber for sleeping, red for cut.
- Technicals (ATR, S20, Stage) pulled via `get_snapshot_metrics` once per
  position. Cache to avoid duplicate Finviz fetches.
- Non-fatal: if SnapTrade fetch fails, write a placeholder page with
  "Last refresh failed at <timestamp>".

**Test:**
- Smoke test in `tests/test_generate_live_portfolio.py` — mock SnapTrade
  + Finviz, run generator, assert HTML contains expected ticker rows +
  verdict tags.

## Open questions for review

1. **T1 auto-peel — half or third?**
   - Spec says half (Minervini canon). Live `/pos-review` skill output today
     also said "peel ½".
   - Alt: ⅓ peel, leaves more runner. Less aggressive locking.
   - Recommendation: **half** for parity with live discipline. Confirm.

2. **Stale-cull threshold — 14 days / +4% peak?**
   - 14 days is conservative — most setups resolve in 5-10 trading days.
   - +4% peak threshold: anything that touched +5% had a real bounce; +4% is
     the "barely showed up" line.
   - Alt: 10 days / +3% peak (more aggressive — better for flying markets)
   - Recommendation: **14 days / +4% peak** for first version. Tunable
     constant — can tighten to 10 / +3% if the rule never fires in normal
     conditions or live book proves a faster decay window is needed.

2b. **Stale paper auto-execute vs live alert-only — both correct?**
   - Paper: auto-sell — system owns the book, free the BP automatically.
   - Live: alert only — user owns execution; system surfaces the decision.
   - Both use identical thresholds + peak calculation. Symmetric logic,
     asymmetric action.
   - Recommendation: yes, both. Confirm.

3. **Live dashboard — generate from position_monitor or standalone?**
   - Inside `position_monitor.py` = one less workflow file, deps already loaded.
   - Standalone script = clean separation, can run on demand outside workflow.
   - Recommendation: **inside `position_monitor.py`** (matches paper pattern).

4. **Live dashboard refresh — every 30 min or only book runs?**
   - Every 30 min = stays fresh during market.
   - Only book runs (3x daily) = lighter, but stale between runs.
   - Recommendation: **every 30 min** — same cadence as critical alerts.
